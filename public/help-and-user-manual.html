Getting Started & How to Use

  1. First, read the README’s Overview section, and then, read the README’s Usage section.
  2. You must read and understand the Disclaimer and the Disclosure linked at the Disclaimer, before proceeding.
  3. If you are running on a Raspberry Pi 5 with Heat Sink and Fan (5GB RAM, and Ethernet cable Internet connection instead of Wi-Fi) or similar device that defaults to a very low-resolution on the screen, consider setting the “Zoom” on your web browser to about 80%, to ease use of the app.
  4. On first load of the app, wait for the initial synchronization of the app with the blockchain data to complete. This initial sync handles the default selected liquidity position, if any. It will be repeated for each brand-new liquidity pool that you have a position on. This is a read-only operation but can take a while. Don’t worry though: The results are cached so that, normally, you won’t have to wait again. While with a valid Moralis API key that is not at its quota limit, it will be fastest, this one-time process can take up to two hours to complete. Watch for the Sync Status badge near top-right on the app to read “Synced”. At that time, the blur on the app details section will clear.
  5. Next, consider which Liquidity Positions you want to manage, by clicking on the “Positions” button near center top on the app if in three column-view, or at the left if in single-column view. Alternatively, you can just manage the currently-loaded position. Before clicking the “Manage” button on any position, check that all settings on the app are appropriate for the gas cost on the blockchain you are using, and for the level of slippage that you will require.
  6. First, check the amount on the “Threshold” setting for Auto-compound. If gas costs are high, you will want this number to be large enough that gas costs are a small fraction of it. Next, scroll down to ensure that the settings under “Bot Configuration” are to your liking. For example, be sure to set a slippage percent that is appropriate for that pair (otherwise, the swaps to rebalance your LP position will fail due to insufficient slippage, or will be too costly if you set slippage too high). Also, if your pair requires high slippage, then you will want to control the frequency of rebalancing for that pair. There are several settings for this and the easiest is the per-day limit. For high-slippage, you might want to restrict rebalances to one per day, to control loss from slippage. Carefully read each circle-i tooltip for all the settings so that you understand them all. When you are ready, click the “Manage” button. Each managed position runs its own independent rebalance loop.
  7. Set Price Range Extension and Position Offset. In Bot Configuration under Range & Execution, set the Price Range Extension percent to control how far the position spans from the current price. If you want an asymmetric token split, adjust the Position Offset; the default 50/50 (No Offset) is right for most pairs. Leaving Price Range Extension blank preserves the position’s existing Range Width on rebalance (see the “Price Range Extension, Range Width, and Position Offset” section below for the difference between the two concepts). To force a full-range rebalance, use the Full-Range checkbox next to the Price Range Extension field. All persist per position and take effect on the next rebalance (manual or automatic). Click the circle-i info icon next to each field for the trade-off details.
  8. Wait for the “Sync” label near top right to finish (“Synced”). Syncing usually just takes a few minutes, but for a very long chain of liquidity positions in a given pool, it can take longer: Almost never more than 30 minutes, but possibly more. Don’t worry though: The results are cached so that, normally, you won’t have to wait again.
  9. When the price moves out of the configured range, the bot automatically rebalances by draining the old position and minting a new one centered on the current price. A rebalance lock ensures only one position transacts at a time (same wallet = same nonce).
  10. Each browser tab shows one position via URL. Open multiple tabs to monitor multiple pools simultaneously. If running on a lighter-weight machine such as a Raspberry Pi 5 with Heat Sink and Fan (5GB RAM, and Ethernet cable Internet connection instead of Wi-Fi) or similar, it’s best to keep it to just one open web browser tab.
  11. Historical rebalance events are scanned up to 5 years back and used to calculate lifetime P&L.
  12. The $usd value of the total lifetime deposit is automatically detected. In case of a problem with that, a dialog pops up to notify the user of the opportunity to enter it manually instead at top left in the app under “Total Lifetime Deposit”. Manual entries there always override any auto-detected value.
  13. Wallet Residuals: Residuals are tokens that you intended to put into a liquidity position but that did not fit the chosen price range. They happen in two places: (1) when you first mint the position on the 9mm Pro dApp — the dApp deposits only what the range can accept at the current price and leaves the rest in your wallet on purpose; and (2) when the bot rebalances into a narrow new range and a little is left over. In both cases LP Ranger recognizes these wallet tokens as residuals for that specific pool, tracked separately from the on-chain LP value. On the very next rebalance the residuals are swept into the freshly minted position and, at that point, roll into your lifetime deposit for the pool. If you would rather not wait, click “Rebalance Now” and confirm the IL warning; the residuals will be folded in immediately. If you want a wider range for that sweep, raise the saved Price Range Extension in Bot Configuration first (or check the Full-Range box for a full-range mint). If multiple positions share a token, the wallet balance is split pro-rata based on each position’s in-range token amount.

FAQ

  • Q: Does LP Ranger charge any fees? A: LP Ranger does not charge any fees of its own, but the blockchain operations involved with providing liquidity do cost gas money.
  • Q: Can I lose money using LP Ranger? A: Generally, it’s the same risk as providing liquidity itself — so, yes, you could lose funds. But as well, although reasonable and commensurate precautions have been taken against this, LP Ranger itself could have bugs in it that could cause loss of funds. Don’t put in more funds than you can afford to lose.
  • Q: Is LP Ranger open-source? A: Yes, completely. See the License section of the README for details.
  • Q: Do I need to leave my computer on for LP Ranger to manage my positions? A: Yes. LP Ranger runs locally on your machine — if it isn’t running, no rebalancing, no compounding, and no notifications happen. LP Ranger runs well on a low-power always-on host like a Raspberry Pi 5 with Heat Sink and Fan (5GB RAM, and Ethernet cable Internet connection instead of Wi-Fi).
  • Q: Is my private key safe? A: Your private key is encrypted at rest with AES-256-GCM using a password you set, and is never sent off your machine. Only you can decrypt it. The plaintext key only exists in memory while the bot is running.
  • Q: Does LP Ranger support Uniswap V2-style positions? A: No. LP Ranger only manages V3 concentrated-liquidity NFT positions. V2 positions are rejected with a clear error.
  • Q: Can I manage multiple LP positions at the same time? A: Yes — LP Ranger is built for this. Each managed position runs its own independent rebalance loop, and a wallet-level lock serializes on-chain transactions so nonces stay clean.
  • Q: Do I need a Moralis API key? A: No, but it’s strongly recommended (free tier is sufficient). Without one, historical price lookups fall back to GeckoTerminal, which is rate-limited and can make the initial sync take significantly longer.
  • Q: If I sign up for a Moralis API key, will I get charged? A: Not if you cap your account at your current tier. Strongly recommended: after creating your Moralis account — whether you stay on the free tier or move to one of the various paid levels — open your Moralis account settings and disable overage / pay-as-you-go billing. Most paid API providers — Moralis included — enable overage charges by default, which means a usage spike that exceeds the quota on your current tier (free, or one of the various paid levels) can result in unexpected bills. With overage disabled, requests beyond your tier’s quota are refused rather than billed, and LP Ranger automatically falls back to GeckoTerminal — so capping the account costs you nothing in functionality. This same recommendation applies to any other optional third-party API key you add to LP Ranger: check each provider’s billing settings and turn overage off wherever it is offered.
  • Q: What happens if my computer crashes mid-rebalance? A: On restart, LP Ranger reloads all running positions from app-config/user-configurable/bot-config.json and resumes polling. If a transaction was pending on-chain when the crash happened, the bot’s speed-up / auto-cancel logic will handle the stuck nonce on the next cycle.
  • Q: Why does the dashboard sometimes show “WITHIN THRESHOLD” instead of rebalancing immediately when out of range? A: The OOR threshold (REBALANCE_OOR_THRESHOLD_PCT, default 5%) gives the price room to wander past the boundary before triggering a rebalance, so you don’t pay gas on every minor wick. The OOR timeout (default 3 hours) eventually forces a rebalance even if the threshold is never crossed.
  • Q: Can I close LP Ranger and re-open it without losing data? A: Yes. Managed positions, settings, P&L history, and the encrypted wallet all persist to disk in app-config/. Closing the browser tab is also fine — the server keeps running until you stop it.
  • Q: Does LP Ranger send any data to a server I don’t control? A: Only what’s needed to read on-chain data, fetch prices, and route swaps — RPC calls to your configured RPC endpoint, price/history calls to DexScreener, GeckoTerminal, or Moralis depending on your settings, and quote requests to online DEX aggregator dApps such as 9mm Aggregator for routing rebalance swaps. Telegram notifications, if configured, go through Telegram’s API. Nothing else leaves your machine.
  • Q: What happens if the RPC provider goes down? A: LP Ranger automatically falls back to another RPC provider for an hour, before switching back to try the main one again.
  • Q: Why didn’t the bot rebalance when I expected it to? A: Several gates can defer a rebalance: the OOR threshold not yet crossed, the minimum interval since last rebalance, the daily cap, volatility doubling mode, the gas-cost guard (defers when gas exceeds 0.5% of position value), or a paused state from a previous slippage abort. Check the Activity Log and the range banner — they explain which gate is active.
  • Q: Why do I see rebalances more frequently than my out-of-range time interval? A: Those extra rebalances are usually re-tries of an initial rebalance, not fresh out-of-range events. There are two common reasons the bot fires a re-try shortly after an initial rebalance: (1) the initial rebalance left a meaningful share of your funds as residual coins in your wallet — tokens that didn’t fit cleanly into the new range — so the bot waits about 10 minutes and then fires an automatic residual cleanup rebalance to fold those leftovers back into the position (see Automatic Residual Cleanup below for the full criteria); or (2) the initial swap had to abort because price impact exceeded your Slippage Tolerance setting, and after the swap-backoff timer expires the bot retries the rebalance. Every re-try still has to pass the Min Time Between Rebalances floor and the Max Rebalances / Day daily cap, so the total work stays bounded. This is intended behavior — the re-tries protect you from leaving funds idle in the wallet or from having the position drift while waiting on slippage conditions to settle.
  • Q: My position is stuck syncing. A: The syncing can be halted from the LP Browser. Click the “Open Position(s)” button at the top right of the header to open the LP Browser. Select the stuck position, then click Remove at the bottom right of the LP Browser. The bot loop ends and the on-chain position is untouched. Re-scan the wallet using the “Scan Wallet” button at the top right of the LP Browser. Optionally, select the position again to resume managing it.
  • Q: What if I can’t find the answer in this FAQ? A: You may check (control-F / command-F (Mac)) the README, the Disclosure, the info at the circled “i” tooltips in the app user-interface, or the code itself. To search the code, sign in to GitHub, and type your search term in the search box at the GitHub Code Search.

Bookmarking Positions

The address bar updates as you switch positions. The URL looks like /pulsechain/<wallet>/<nft-contract>/<tokenId>. Add it to your browser bookmarks (or drag the address bar icon to the bookmarks bar) to jump straight to any position later. Opening a bookmark loads LP Ranger and activates that exact position as soon as the app is ready. Each bookmark is self-contained, so you can keep one per pool or one per NFT and switch between them with a single click. Bookmarks survive app restarts because they only reference on-chain identifiers — nothing in the URL depends on the server’s current managed-positions list.

All Positions Stats

LP Ranger lets you view the performance of all your LP positions at once, so you can see what’s working and what might need adjustment.

The All Positions Stats button in the header (right of the Open Positions button) opens a sortable table across every currently-open managed position on your wallet. Columns: identity (blockchain, token pair, fee tier, LP provider), Lifetime P&L, Profit, Impermanent Loss/Gain, and a per-row Position link that takes you to that position’s dashboard tab.

Click any of the three numeric column headers to sort. First click sorts descending; a second click toggles ascending. The default sort is Lifetime P&L descending. Rows re-render on the same 3-second poll as the rest of the dashboard, so numbers stay live while the modal is open.

The button stays disabled until every currently-running managed position has finished its rebalance-history and lifetime-deposit scans. Hover the disabled button for a tooltip showing how many positions are still loading. For the meaning of each numeric column, click the “i” info icon on the Lifetime panel of any managed position; the same formulas power the modal.

Mission Control

Mission Control is the center card on the dashboard. It houses the manual action buttons and the auto-compound controls for the currently-viewed managed position.

  • Special Action badge at the top of the panel shows what the bot is currently doing on this position: idle (“Special Action: None”), rebalancing, compounding, or a residual cleanup in progress. If multiple actions overlap (rare), each stacks inside the badge.
  • Rebalance Now triggers a manual rebalance. Clicking it opens a short impermanent-loss confirmation dialog before firing. The rebalance uses the Price Range Extension, Full-Range checkbox, and Position Offset saved in Bot Configuration → Range & Execution for this position; there is no per-rebalance width prompt.
  • Compound Now collects any unclaimed fees and redeposits them into the current position. The button enables only when unclaimed fees are worth more than the Compound Min Fee gate (defaults to $1, tunable in .env). No new NFT is minted; the same NFT gains liquidity.
  • Auto-compound toggle: when on, the bot compounds automatically once unclaimed fees cross the Threshold you set below the toggle. The Threshold input takes a USD value; anything above it triggers a compound on the next in-range poll cycle. Both persist per position.

For deeper trade-offs on when to compound vs. rebalance, and on how the Threshold interacts with gas cost, click the “i” info icons next to Auto-compound and Threshold.

Fund Tracking

When a position is rebalanced, all collected tokens (principal + earned fees) are recycled into the new position. Any remainder that doesn't fit the new range stays in your wallet and is automatically counted toward your P&L at current market prices, capped to your actual wallet balance. If you withdraw or sell tokens yourself, use Edit Realized Gains to account for them so your P&L stays accurate. Click the “i” info icon next to Realized Gains for the exact rules on how the field interacts with wallet residuals.

Automatic Residual Cleanup

After every successful rebalance, LP Ranger waits 10 minutes and then checks whether the wallet still holds a meaningful residual share of the position’s tokens. If the residual is still more than 5% of the combined LP + residual value for that pool, the bot fires one additional rebalance to sweep the leftover tokens into the position. This is called an automatic residual cleanup.

When it triggers. A cleanup arms itself only when every one of the following is true:

  • At least 10 minutes have elapsed since the last rebalance.
  • The residual wallet share exceeds 5% of (LP + residual) for the pool.
  • No rebalance is currently in progress, paused, or already requested by you.
  • A cleanup has not already been consumed since the last normal rebalance (see “Back-to-back protection” below).

Back-to-back protection. A cleanup may fire at most once per normal rebalance. If the cleanup itself does not reduce the residual below 5% — for example, because liquidity at the edge of the range is too thin — the bot does not keep trying. The cleanup opportunity is reset only when the next out-of-range or manual rebalance succeeds. This prevents the bot from oscillating on pools where a small residual is structurally unavoidable.

How the sweep decides it’s done. The cleanup rebalance runs the same execution as any other rebalance: remove liquidity, swap, corrective-swap loop, mint. The corrective-swap loop uses LP Ranger’s inflation-resistant dust threshold as its exit condition. That threshold is gold-pegged (via a PAXG / XAUT reference in dust-threshold.json) instead of USD-pegged so it does not creep up with fiat inflation. The 5% pool-share is only what arms the cleanup; the actual sweep continues until residual imbalance falls below the dust threshold or the corrective-swap iteration cap is reached.

All the usual gates still apply. A cleanup still respects the throttle (minimum interval, volatility doubling), the daily rebalance cap, the swap-backoff timer, the paused state, and dry-run mode. The cleanup is simply a reason to ask for a rebalance — every safety rule downstream continues to govern whether and when it actually runs.

What you’ll see. While a cleanup rebalance is in flight, the range banner at the top of the position card turns yellow and flashes, reading “Rebalancing to Reduce Residual Wallet Coin Amount”. The resulting entry in the Activity Log is tagged “(Residual Cleanup)” so you can distinguish it from an out-of-range or manual rebalance at a glance. There is no separate Telegram alert — cleanup rebalances use the same “Rebalance Succeeded / Failed” channels as any other rebalance.

If you don’t want to wait. You can always click Rebalance Now to sweep residuals immediately. Confirm the IL warning that appears and the rebalance fires with the Price Range Extension, Full-Range checkbox, and Position Offset you have saved in Bot Configuration. If you want a wider range for that rebalance, increase the saved Price Range Extension first (or check Full-Range), then click Rebalance Now.

Tuning. The 10-minute delay and 5% threshold ship in app-config/app-defaults-for-user-configurable/bot-config-defaults.json under the residualCleanup group (fields delayMs and thresholdPct). They are intentionally not exposed in the Bot Configuration UI — they are cross-cutting defaults that apply to every managed position. To override, copy the file app-config/app-defaults-for-user-configurable/bot-config-defaults.json to app-config/user-configurable/bot-config-defaults.json, and edit your copy to set just the residualCleanup keys you want to change (you do not need to copy the rest — the runtime deep-merges your copy on top of the shipped defaults, so any key you omit keeps its shipped value). Your copy in user-configurable/ survives tarball upgrades; the original in app-defaults-for-user-configurable/ must stay untouched.

Price Range Extension, Range Width, and Position Offset

These live in Bot Configuration under Range & Execution, persist per position, and take effect on the next rebalance (manual or automatic). No forced rebalance is needed to change any of them.

Price Range Extension vs Range Width — two different concepts

LP Ranger distinguishes two related but different concepts for “how wide is the position”. Only the first is implemented today; the second is a future feature on the road map.

  • Price Range Extension (implemented): how far the position’s price range extends from the current price, expressed as a percentage of the current price. At the default 50/50 offset, a Price Range Extension of 50 means the position covers 25% below current price and 25% above current price — so from 0.75× current to 1.25× current. A value of 10 covers ±5% around current (0.95× to 1.05×).
  • Range Width (future / on the road map, see the “Nice to Have’s” list): the percentage of the pool’s currently-populated liquidity range that the position covers. This is the LP-native concept — “how much of where liquidity actually exists am I in?”. It is more useful for deciding where to place coins because it accounts for the pool’s real liquidity distribution rather than an abstract percentage-of-current-price. It is not yet implemented; the app documents it here so the terminology is consistent with future work.

Different LP creators (for example the 9mm dApp’s own position UI) may label their width slider using yet a third convention — linear-vs-exponential math, symmetric-in-price vs symmetric-in-tick, and so on. A position you set up as “50%” on 9mm may not read as “50” in LP Ranger’s Price Range Extension field, because the two systems use different denominators. That is expected; LP Ranger is self-consistent internally.

When Price Range Extension is left empty (unset)

Leaving the Price Range Extension field empty tells the bot to preserve the existing Range Width on every rebalance — the position’s on-chain tick spread is carried across the rebalance and re-centered on the current price. It is subject only to standard liquidity-pool tick rounding, which can drift the spread by up to one tick-spacing per rebalance. The input stays blank until you type an explicit value — it does not display any auto-computed fallback (that was misleading; it looked like a saved value when the user had not set one).

Full-Range checkbox

To the right of the Price Range Extension field is a Full-Range checkbox. When checked, every rebalance mints a full-range position (from the pool’s minimum tick to its maximum tick — effectively “price zero to infinity”), and the Price Range Extension value is ignored. The field is disabled while the box is checked. If the currently-active position is already full-range on-chain, the box shows as checked automatically, reflecting on-chain reality, even without an explicit save.

Edit-row buttons

  • Default: fills the input with the shipped default from bot-config-defaults.json. You still have to click Save to persist it.
  • No Override: clears any saved override. The input goes empty and the config field is set to null. Rebalances then use the “preserve the existing Range Width” behavior described above.
  • Save: writes the value in the input to the per-position config; applies on the next rebalance.

Position Offset

The Position Offset controls how the Price Range Extension is split above versus below current price. At the default 50/50 (No Offset), the range is centered on current. At 40/60 or other skewed values, more of the range sits on one side, giving you a more asymmetric token composition on rebalance.

For the full trade-off explanation (fee capture vs. impermanent loss, out-of-range behaviour, etc.), click the “i” info icon next to each field in the app.

Slippage Tolerance (per-token)

Slippage tolerance is set independently for each token in the pair. This handles asymmetric-liquidity pairs cleanly: one side may be thinly traded and need a high slippage to complete swaps, while the other side is deep and needs a low value to avoid opening its swaps up to MEV front-running. A single slippage value could not do both jobs at once; per-token slippage does.

Both inputs live in Bot Configuration → Range & Execution, labelled Slippage (<token 0 name>) and Slippage (<token 1 name>). Token names update automatically to match the active pool.

Default value

Both inputs start populated with the shipped default (0.75%). That is the value the bot uses on that side until you change it. Type a different value in either input and click Save to persist it for this position. On upgrade from a version that only had one Slippage Tolerance input, the previously saved value is not migrated — both per-token inputs start at the shipped default. You can re-tune them as needed.

Which side applies to which swap

The bot uses the destination-token rule: a swap that CONVERTS Token 0 INTO Token 1 uses the Slippage (Token 1) value, and a swap that CONVERTS Token 1 INTO Token 0 uses the Slippage (Token 0) value. The destination side is where MEV can extract value from the swap, so that’s where the slippage budget lives.

Allowed range and warnings

The full allowed range per side is 0.1% to 20%. When you Save a value above 5%, the app opens a confirm dialog before writing it. When you Save a value at 10% or higher, the app opens a stricter dialog that requires you to type Confirm (case-sensitive) into a text box before it proceeds. Values outside 0.1% to 20% are rejected and the app shows a warning stating the valid range. On asymmetric pairs, the deep side typically wants 0.1% to 1% and the thin side may need 5% or more.

Volatility Doubling

If 3 rebalances occur within 4× the minimum interval (e.g. 40 minutes when the minimum is 10 min), the bot enters doubling mode. Each subsequent rebalance must wait twice as long as the previous one: 10m → 20m → 40m → 80m, and so on. This prevents rapid-fire rebalancing during high volatility. Doubling mode clears automatically if no rebalance is needed for 4× the current wait, or at the daily midnight UTC reset.

Gas Cost Protection

Before each rebalance, the bot estimates gas cost in USD. If gas would exceed 0.5% of the LP position value, the rebalance is deferred and the bot retries once per hour instead of once per minute. Normal polling resumes automatically once gas drops or a rebalance succeeds. The bot always starts with the normal polling interval.

Closed Positions

When you select a closed position (zero liquidity), the bot will display its data but will not attempt to rebalance it. If you re-open a previously closed position by adding liquidity back to the same NFT on 9mm Pro, the bot treats it as brand new—duration, fees, and P&L all start from the original mint date. Recommendation: instead of re-opening a closed position, mint a fresh LP position on 9mm Pro. This gives you a clean start with accurate tracking and avoids any ambiguity in your P&L history.

Telegram Notifications

LP Ranger can send real-time alerts to your phone via Telegram. You choose which events trigger a notification:

  • OOR Timeout Triggered (default ON) — the position has been continuously out of range for longer than the configured timeout, and a rebalance is about to begin.
  • Rebalance Succeeded (default OFF) — a rebalance completed successfully and a new NFT was minted.
  • Rebalance Failed (default ON) — a rebalance attempt failed (on-chain revert, RPC error, etc.).
  • Compound Succeeded (default OFF) — unclaimed fees were successfully compounded back into the position.
  • Compound Failed (default ON) — a compound attempt failed.
  • Other Error (default ON) — an unexpected error occurred during a poll cycle, including stuck transactions that were auto-cancelled. Does not include server/bot shutdown, which has its own event below.
  • Low Gas Balance (default ON) — the wallet’s native token balance (e.g. PLS) has dropped into the amber “top up soon” zone: below the recommended cushion (one worst-case rebalance × 3× safety × number of managed positions) but still above the one-rebalance floor. The alert fires once per low-balance episode and resets when the balance recovers above the recommended threshold, so you won’t be spammed.
  • Very Low Gas (default ON) — the wallet’s native token balance has dropped below the one-rebalance floor and the Mission Control gas badge is flashing red. The next rebalance will fail on gas. This alert fires once when the tier drops to critical and resets only when the balance recovers to the “ok” tier.
  • Server and Bot Shutdown/Exit (default ON) — the server (which includes the bot) is shutting down, whether via the dashboard, npm run stop, or a SIGINT/SIGTERM signal. A manual restart may be required.

Setup:

  1. Open Telegram and message @BotFather. Send /newbot, follow the prompts, and it will give you a bot token (looks like 123456789:ABCdefGhI...).
  2. Save the bot token to a safe place. You will need to copy it soon.
  3. Start a chat with your new bot and send any message (just type something and press Enter).
  4. In your web browser, go to: https://api.telegram.org/bot<your-bot-token>/getUpdates (replace <your-bot-token> with the token from Step 2, no brackets).
  5. In the JSON response, find your chat ID at result[0].message.chat.id — it is a numeric value near the top of the response, inside the first result entry, under message → chat → id. Copy this number.
  6. In LP Ranger, open the Telegram setup dialog from the Settings gear menu or from the wallet setup screen.
  7. Paste your bot token and chat ID, select the events you want, and click Save.
  8. Click Test to verify — you should receive a test message in Telegram.

Your bot token is encrypted on disk with the same wallet password used for your private key. It is never sent anywhere except the Telegram Bot API.

Troubleshooting

  • Check Sync Status: Does the “Sync” status at top right show “Synced”? If not, wait for it. Data and controls may be incomplete until the initial blockchain scan finishes.
  • Rebalance Not Completing: If the blockchain is busy, a rebalance could take longer than expected. The progress of a rebalance can be seen on the Activity Log of this app, where the creation of the new NFT will be seen logged as a distinct log item. Also, it can be monitored by viewing the server console log.
  • Stuck Transaction Auto-Cancel: If a rebalance transaction does not confirm within 2 minutes, the bot automatically resends it with higher gas (1.5× bump). If the transaction is still stuck after 20 minutes total, the bot cancels it by sending a zero-value self-transfer at the same nonce, freeing the wallet for future transactions. Each phase is reported in this app’s Activity Log and on the server console. These timeouts are configurable via TX_SPEEDUP_SEC (default 120) and TX_CANCEL_SEC (default 1200) in the .env file.
  • Persistent Transaction Problems: If transactions continue to fail or get stuck, try switching to a different RPC endpoint using the RPC URL control toward the bottom of the app under Bot Configuration.
  • Price Discovery Failures: If USD price lookups are failing (DexScreener is the default), you can configure alternative price services such as DexTools by setting the appropriate API key in your .env file. See the detailed documentation in the file-header of server.js for all available pricing options.
  • Prices Don’t Look Right: Try adding your Moralis API key (free) so that Moralis can be a data source. Moralis is a remotely-hosted crypto data API that provides token prices, historical block-level pricing, and other on-chain data. It is the most reliable price source for tokens that other free services miss. Click the gear icon (top right) to add your key in Settings.
  • Slow Initial Scans: The first blockchain scan fetches historical prices for every rebalance epoch. Adding a Moralis API key (free) makes this much faster by bypassing GeckoTerminal rate limits. Click the gear icon (top right) → Settings.
  • Lost Managed Positions or Settings: On every startup the app saves a backup of your configuration to app-config/user-configurable/bot-config.backup.json. If your managed positions or settings are lost, copy the backup over the main file: cp app-config/user-configurable/bot-config.backup.json app-config/user-configurable/bot-config.json and restart. See the “app-config/” section in the file-header documentation of server.js for the full layout.
  • Current or Lifetime Panel Not Populating: If the Current and/or Lifetime panels are missing data or look stale, re-select the position you want to view from the LP Browser. This triggers a fresh fetch and redraw for that position.
  • Position Stuck on “Syncing…”: If the sync badge at top right stays on “Syncing…” for longer than expected (typically about 30 minutes, occasionally more on very old pools and under slow external conditions) and appears wedged — no progress in the Activity Log, no new entries in the server console — the position’s initial load can get abandoned by a transient network hiccup — for example, the CSRF token going stale during a Chrome socket-pool saturation on a low-powered host. First try a browser page reload — this tears down the tab’s HTTP sockets and fetches a fresh CSRF token, which is often enough to let the already-running server-side scan report its result. If the badge is still stuck after the reload, click the Settings gear at top right, then click Reload Current Position. See the Reload Current Position entry below for what that button does and how long it takes.
  • Reload Current Position (Settings → Reload Current Position): Wipes every on-chain-derived figure for the currently-viewed position (compound history, HODL baseline, deposits, cached epochs, cached event log) and re-runs the initial scan from pool creation. Use it when Lifetime figures look wrong for a specific position (fees compounded, IL/G, deposit total) or when a red “Catastrophic scan failure” dialog surfaces (see Where do error details go? below). While the scan runs, LP Ranger shows a full-screen blocking dialog and disables the rest of the UI; when the scan finishes, the page automatically reloads. Duration ranges from a few minutes up to about four hours, depending on how many rebalances the position has and how responsive your RPC endpoint is. The button is automatically disabled while the current position is mid-rebalance or mid-compound; wait for that operation to finish before clicking. Only affects the currently-viewed position; other managed positions are untouched.
  • Where do error details go? When a catastrophic scan failure occurs, LP Ranger writes the full stacktrace and timestamp to logs/error.log in your installation directory (same folder as the diagnostic lp-ranger.log). Each incident is separated from the previous one by four blank lines so entries are easy to spot when the file is opened weeks later. This file is reserved for catastrophic failures only; routine warnings and info are not written here. Include the relevant section when reporting a bug.
  • Web browser seems stuck while trying to load a new Liquidity Position: If it’s been over an hour, then click the Web Browser’s reload button. This is necessary if the Web Browser’s Console Log shows a “403 Forbidden” error(s). No need for a hard refresh; just click the refresh button.
  • The $usd totals for my LP Position in Unmanaged don’t seem to match up with those shown in the Managed state: Unmanaged shows everything at today’s prices to keep the view fast and simple. Managed records each compound at the price it actually happened, so its totals can drift slightly when prices have moved since. The numbers should still be close. To see the exact bot-tracked figures, click Manage.
  • Turning Sounds Off: LP Ranger plays short audio cues on successful rebalance, successful compound, and when you click “Manage Position.” To silence them, click the Settings gear icon at top right and uncheck Sounds. The choice is saved in your browser and survives restarts. Re-check the box to turn sounds back on. Sounds default to on for a fresh browser profile. To flip the operator-configured default to off for any user who clears their browser storage or visits for the first time, copy the file app-config/app-defaults-for-user-configurable/ui-defaults.json to app-config/user-configurable/ui-defaults.json, and edit your copy to set "soundsEnabled": false. Your copy in user-configurable/ survives tarball upgrades; the original in app-defaults-for-user-configurable/ must stay untouched.
  • Compounds Missing from Activity Log: Compound events are only recorded in the Activity Log if the app was running at the time the compound took place. Historical compounds that occurred before the app was started are not currently surfaced in the Activity Log (they are still counted in P&L and the “Fees Earned” KPI).
  • Why didn’t all my fees get compounded on the latest compound? A compound runs a small ratio-correcting swap before the deposit, so in most cases nearly all of the fees end up in the position. The swap is skipped, however, when (a) the imbalance is too small to be worth swapping (the gold-pegged dust gate) or (b) the gas cost would exceed 1% of the swap value (the gas gate). When either gate trips, the compound deposits only the side that fits the current ratio and leaves the rest as a wallet residual, to be folded back in on the next rebalance (manual or automatic). Note that when the swap fires, the deposit uses the post-swap wallet balance directly. In practice this only sweeps in the tiny amounts that arise from rounding precision and from the small difference between the swap amount the bot projected and the slightly different amount the swap actually returned — not your accumulated wallet residuals from prior rebalances. Those larger residuals are still cleared by the next rebalance, which always sweeps in every wallet residual for the pool when it mints the new position.
  • Why does my compound notification sometimes show “reinvested” greater than “collected”? This is correct behavior, not a bug. When the ratio-correcting swap fires during a compound, the compounder uses the wallet’s post-swap balance directly — which includes any pre-existing residual tokens that were already sitting in the wallet from prior rebalances or compounds. So a log line like collected $5.14 reinvested $6.65 residual $0.00 means the bot put $6.65 of value into the position by combining the newly-collected $5.14 in fees with roughly $1.50 of leftover wallet residual that fit the current ratio range. The “reinvested” figure represents what actually got deposited; “collected” is only the newly-collected fees from this compound cycle. Old residuals getting put back to work is a feature — it cleans up dead tokens from prior moves without waiting for the next out-of-range or residual-cleanup rebalance.
  • Native Gas Token Wraps: EVM liquidity pools only accept the wrapped form of the native gas token (for example, $wPLS on PulseChain — the wrapped equivalent of native $PLS). LP Ranger detects each wrap as fresh capital by scanning the wrapped-token contract’s on-chain Deposit event, which records the exact wrapped amount minted to the wallet. The Deposit event is emitted whether the wrap is a plain wallet action or bundled inside a swap or rebalance, and whether or not the gas fee is netted from the wrap output, so wraps are credited at full precision with no tolerance fudge.
  • Need to Capture a Long Session of Server Logs? Start the app with the --log-file flag and every line that would scroll past in the terminal is also appended to a file that is located at <installation-root>/logs/lp-ranger.log. Usage: npm start -- --log-file (uses the default path above) or npm start -- --log-file path/to/run.log (custom path). The -- separator tells npm to forward the rest of the line to the app. For an always-on tee — useful on a long-lived headless host — copy the file app-config/app-defaults-for-user-configurable/logging.json to app-config/user-configurable/logging.json, edit your copy to set "enabled": true, and run npm start normally (no flag needed). Your copy in user-configurable/ is gitignored, so it survives tarball upgrades; the original in app-defaults-for-user-configurable/ must stay untouched. ANSI color escapes are stripped from the file so it’s grep-friendly; the terminal still shows colors. The file is opened in append mode — rotate or truncate it externally if it grows unbounded. To delete the file (free disk, start a clean capture, or scrub before sharing), stop the app and then run npm run clean:log.

Details

  • Gas Costs: All crypto gas costs displayed in this app are computed using the current price of the native gas token (e.g. PLS on PulseChain). This means gas figures always reflect what the gas would cost at today’s price, not the price at the time of the original transaction.
  • Historical Prices: Historical token prices (from GeckoTerminal, Moralis, etc) are used in two ways: (1) to show the original value of the total lifetime deposit into an LP position at the time it was created, and (2) to show the original value of the first historical LP position deposit for the purpose of calculating lifetime gain/loss over a five-year lookback on the current wallet.

Edit Initial / Lifetime Deposit

LP Ranger automatically detects the USD value of your initial deposit by looking up historical token prices at the time you minted the position. This auto-detected value appears in the Lifetime panel as the deposit amount. However, historical prices are not always available (especially for newer or low-liquidity tokens), so the app lets you enter the deposit value manually.

How it works:

  • If you enter a value and save, your manual entry takes priority over the auto-detected value. The app will never overwrite your manual entry.
  • The auto-detected deposit from the blockchain scan runs independently in the background. It does not change your saved value — it only serves as a fallback when no manual value is set.
  • To revert to auto-detection, clear the input field (leave it empty) and click Save. This removes your manual override, and the app will use the auto-detected value on the next refresh.

A small info icon next to the deposit value tells you which source is being used: “Manually entered”, “Valued using Historical Price”, or “Valued using Current Price” (when historical prices were unavailable and the current price was used as a fallback).

Edit Total Lifetime Days

The Lifetime panel shows how many days the pool has been active for you, and that number drives every APR figure in the panel. Click Edit Total Lifetime Days to enter a value manually.

How it works:

  • Your manual entry wins outright. If you save a value, it overrides every auto-detected date. Type the number of days you want to see right now; tomorrow the display will read one higher, the day after that one higher again, and so on — you never need to re-edit as time passes.
  • To revert to auto-detection, clear the input (leave it empty) or save 0. The app will fall back to the auto-detected start date on the next refresh.
  • Auto-detection picks the mint date of the oldest NFT (for the current liquidity pool) on the current wallet.
  • If that NFT was originally minted on a different wallet and later transferred here, the app uses that NFT’s true mint block (one extra RPC lookup) instead of the arrival date, so time spent on the previous wallet still counts toward Lifetime Days. Only the single oldest NFT gets that lookup — the app does not spider into other wallets’ histories.
  • Edit manually whenever auto-detection is off — for example, if the NFT was minted more than 5 years ago (outside the on-chain scan window) or spent time on multiple prior wallets that the app does not spider.

Checking for Updates

LP Ranger checks GitHub for newer releases every time you open the About dialog. Click the Settings gear icon at top right in the app, then click About.

The About dialog shows the commit hash and commit date of your running copy. If a newer official release is available, a line appears below the commit row reading “Update available: X.Y.Z — Get the update”. The Get the update link opens that release’s GitHub page in a new tab. If the line reads “Up to date”, you are already on the latest release.

To retrieve and install an update:

  1. Click Get the update in the About dialog, then download the release’s .tar.gz asset from the GitHub page that opens.
  2. Follow the Update instructions in the project README: https://github.com/nottoseethesun/lp-ranger#update

If the About dialog does not show an update notice but you believe one is available, confirm you have network access to api.github.com and reload the app.
